## 1. Цель и scope

### Цель
Построить аналог [recyclemap.ru](https://recyclemap.ru) только для Санкт-Петербурга: карта пунктов приёма вторсырья с фильтрами, деталями, маршрутом, формой «Предложить точку», модерацией. С настоящей БД и автоматическим импортом данных.

### В scope
- Карта 300–500 точек приёма, импортированных из OSM + Раздельный Сбор + ручные добавления
- Фильтр-чипы по 8 категориям + время работы + доступность
- Кластеризация, URL-стейт, стабильный URL для точки, share-кнопка
- Сайдбар деталей (desktop) / bottom-sheet (mobile)
- Встроенные маршруты Яндекс + кнопка Google Maps
- Форма «Предложить точку» без регистрации, с hCaptcha
- Очередь модерации с превью на карте
- Email-уведомления админу (новые предложения) и пользователям (одобрение/отказ)
- Админ-только авторизация через Яндекс OAuth + whitelist
- Автоматический weekly-импорт OSM

### Не в scope (backlog / потом)
- Регистрация публичных пользователей, профили, избранное
- Комментарии, голосование, рейтинги
- Мульти-город (пока только СПб)
- Мульти-ролевая модель (пока один админ = вы; модераторы появятся позже если понадобится)
- Апдейты через чат-бот (Max-bot / VK Mini App / Telegram) — [записано в backlog](../backlog.md)
- Донаты, монетизация
- PWA / install prompts

### Аудитория и контекст
- Жители Петербурга, ищущие куда сдать вторсырьё
- Мобильный трафик ожидается основным
- Интерфейс только на русском языке

## 4. Ключевые флоу

### 4.1. Посетитель смотрит карту
1. Открывает `recyclemap-spb.ru` — Server Component читает YDB напрямую (creds серверные) → отдаёт HTML с уже встроенными точками
2. Leaflet рисует маркеры, `react-leaflet-cluster` группирует близкие
3. Клиентская обёртка (HomeClient) держит фильтр-стейт (useState), пересчитывает видимые точки при смене категорий
4. Смена фильтра-чипа → список в памяти пересчитывается (useMemo), визуально обновляется на карте
5. Клик по маркеру → bottom-sheet (mobile) / сайдбар (desktop) открывается с деталями, URL становится `/point/p001`

**Почему через Server Component:** YDB-креды хранятся в env на сервере и не покидают runtime — клиент видит только готовый HTML с данными.

### 4.2. Посетитель предлагает точку
1. Кнопка «Предложить точку» в шапке → `/submit`
2. Форма: адрес с автокомплитом Яндекс Геокодера, чипы категорий, часы, телефон, сайт, описание, email, hCaptcha
3. Submit → `POST /api/submissions` → валидация → INSERT в `submissions` со `status='pending'`
4. Email админу («новое предложение») + подтверждение отправителю
5. Страница «Спасибо, мы свяжемся»

### 4.3. Админ модерирует
1. `/admin/moderation` → NextAuth redirect на Яндекс OAuth
2. Callback → NextAuth проверяет `admins` → допуск или `403`
3. Таблица pending + карта-превью каждой
4. «Одобрить» → INSERT в `points` + `point_categories`, `submissions.status='approved'`
5. «Отклонить» → `status='rejected'` + `admin_note`
6. Email отправителю (одобрено / отклонено с причиной)

### 4.4. Крон импорт OSM
1. YC Cloud Function Trigger: воскресенье 03:00 МСК
2. Функция: Overpass API → `(node["amenity"="recycling"](59.8,29.5,60.1,30.6);); out body;`
3. Маппинг: теги → наши категории (`recycling:plastic=yes` → `plastic`)
4. Upsert по ключу `(source='osm', source_id)`. Если `manually_edited=true` — не трогаем поля.
5. Email админу: «+3 новых, 12 изменений»

**Никогда не удаляем точки автоматически** — админ сам переводит в `status='closed'`.

### 4.5. Админ вручную добавляет/редактирует точку
1. `/admin/points/new` — форма с теми же полями что в `/submit`, но без капчи и с прямой вставкой
2. `/admin/points/:id/edit` — редактирование, ставит `manually_edited=true` на изменённые поля

## 5. UI/UX

### Десктоп (≥768px)
Три колонки: **фильтры (280px) | карта (flex-1) | сайдбар деталей (340px, по клику на точку)**.

В шапке: логотип + кнопка «+ Предложить точку» + ссылка «О проекте».

### Мобильный (<768px)
- Карта на весь экран
- **Bottom-sheet** (vaul) в двух состояниях:
  - **Collapsed:** узкий язычок «18 точек на карте | Фильтры ▴»
  - **Expanded:** чипы категорий + доп-фильтры + счётчик + «Очистить (N)»
- Клик по маркеру → bottom-sheet заполняется деталями точки (вместо фильтров)

### Чипы категорий
- Компактные (padding 6–10px, border-radius 16px, font-size 12px)
- Выбранная: фон = `category.color`, белый текст
- Невыбранная: светло-серый фон, границ, тёмный текст
- На мобильном — flex-wrap в две строки
- Клик по чипу = toggle, URL обновляется

### Сайдбар деталей точки
- Название + адрес + крестик закрытия
- Бэйджи всех категорий точки (цветные, с эмодзи)
- Кнопки: «🗺 Яндекс» (красная, primary) | «📍 Google» | «🔗» (share)
- Часы, сайт, телефон (как ссылки `tel:` / `href`)
- Описание в светло-сером прямоугольнике

### URL-стейт
```
/?fractions=plastic,glass&lat=59.94&lng=30.31&zoom=11
/point/p001?fractions=...   (открытая точка сохраняет фильтры)
```

### Темы и доступность
- Тема: только светлая (dark mode — позже)
- ESC закрывает сайдбар/bottom-sheet
- Tab-навигация по чипам и кнопкам
- Русский язык (`lang="ru"`)

## 6. Импорт данных

### Источники (по приоритету)

**1. OpenStreetMap Overpass API** — автомат, основа
- Запрос: `node["amenity"="recycling"]` в bbox СПб
- Ожидаемо: 300–500 точек
- Надёжно: координаты, адрес; редко: часы, контакты

**2. Раздельный Сбор (rsbor.ru)** — скрейпинг их каталога
- ~50 проверенных мест с богатыми данными
- Первый прогон — все через модерацию (админ одобряет руками)

**3. Ручные добавления админа** — особые места

### Ключ дедупликации
UNIQUE `(source, source_id)`. При совпадении — UPDATE; иначе — INSERT. Поля с `manually_edited=true` импорт не трогает.

### Pipeline (YC Cloud Function)
```
YC Cloud Function (cron еженедельно)
 ├── pull from OSM Overpass
 ├── pull from rsbor.ru (scrape HTML)
 ├── dedup по (source, source_id)
 ├── upsert в YDB (points, point_categories)
 └── email-отчёт админу
```

### Принципы
- **Никогда не удаляем точки автоматом.** Админ решает, закрывать или нет.
- **Не перезаписываем manually_edited поля.** Автоимпорт уважает ручные правки.
- **Первый запуск — вручную.** Мы запустим функцию сами и проверим результаты.

## 8. Миграция с текущего MVP

Текущий MVP (Next.js + Leaflet + JSON + Vercel, 30 точек, рабочие фильтры-чекбоксы) — **не выбрасываем, а постепенно превращаем в целевой продукт**.

**Что остаётся нетронутым:** Next.js структура, TypeScript-типы, Tailwind, CI-пайплайн в репо.

**Что меняется поэтапно:**

| Было | Станет | Когда |
|---|---|---|
| JSON `data/points.json` | YDB таблица `points` | Phase B→C |
| Leaflet + OSM тайлы | Яндекс Карты JS API 3.0 | Phase C |
| Checkbox-фильтры | Chip-фильтры | Phase D |
| Popup на карте | Сайдбар / bottom-sheet | Phase D |
| Нет формы | `/submit` + hCaptcha | Phase E |
| Нет админки | `/admin/*` | Phase F |
| Нет авторизации | NextAuth + Яндекс OAuth | Phase A |
| Vercel | YC Serverless Container | Phase A |
| Нет крона | YC Cloud Function + Trigger | Phase B |

Репозиторий `github.com/victorovi4/recyclemap-spb` остаётся тем же. Ветка `main` — основная. Для каждой фазы — ветка `phase-a`, `phase-b` и т.д., мержим в main по готовности.

## 9. Риски и открытые вопросы

| Риск | Вероятность | Чем закрываем |
|---|---|---|
| Квота YDB-баз (4/4 была) | закрыта | удалили `gdegrant-test`, создали `recyclemap-spb` |
| Rsbor.ru поменяет HTML-структуру → скрейпер сломается | средняя | мониторить через email-отчёт; на падении — откладываем до починки |
| OSM-данные неровные по качеству | высокая | модерация помечает `manually_edited`, ручные правки переживают импорты |
| Email-webhook из GdeGrant не подойдёт под наши нужды | низкая | запасной план: Resend (100/день бесплатно) |
| NextAuth v5 + Яндекс-провайдер — не было примеров | низкая | Яндекс-провайдер есть в NextAuth из коробки, проверим на Phase A |
| Потеря данных при миграции с JSON | минимальная | 30 точек небольшие, перезальём в YDB при первом импорте |

### Открытые вопросы (решим при старте фаз)
- **Домен:** покупаем `recyclemap-spb.ru`? (~800₽/год) или оставляем YC-поддомен? → решаем в Phase G
- **Аналитика:** Яндекс.Метрика или Posthog? → решаем в Phase G
- **Модераторы (не только админ):** нужно ли добавить вторую роль? → добавим отдельной фазой если понадобится
- **Фото точек:** включаем Object Storage + upload с первого релиза или позже? → **позже** (backlog)
